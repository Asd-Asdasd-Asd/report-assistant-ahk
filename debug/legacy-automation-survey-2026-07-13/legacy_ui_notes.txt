Survey conclusion
------------------------------------------------------------
The montage workflow contains mixed UI technologies:
- Custom-rendered controls requiring relative clicks.

The production implementation should evaluate each step independently rather
than forcing the entire workflow into either pure coordinate automation or pure
UIA automation.

UI target: Screenshot button
============================================================
Related action:
capture-current-viewer-image
Known legacy click:
MouseClick "left", 1981, 1350
Observed UIA properties:
- Name: Property does not exist
- ControlType: Button (50000)
- LocalizedControlType: 按钮
- IsKeyboardFocusable: True
- BoundingRectangle: [l=1934,t=1314,r=2028,b=1390]
Accessibility interpretation:
- The screenshot target is exposed as a standard UIA Button.
- It has no useful accessible Name.
- The surrounding interface contains multiple unnamed Button elements.
- Identifying it only as the first unnamed Button is unsafe.
Available activation methods:
1. Official keyboard shortcut F12.
2. UIA Invoke after reliable geometric or parent-container identification.
3. Click inside the UIA Button rectangle.
4. Absolute coordinate click as legacy fallback.
Recommended priority:
1. F12.
2. UIA Invoke if a need for explicit button activation remains.
3. Button-rectangle-relative click.
4. Absolute coordinate only for diagnostics.
Reason for preferring F12:
- It is an application-defined shortcut.
- It is independent of monitor coordinates and layout.
- It avoids ambiguity among multiple unnamed buttons.
- It has already been observed to work reliably.
Feedback requirement:
The legacy mouse excursion provided execution feedback.
When moving to F12, replace this with a non-focus-stealing indicator rather
than retaining coordinate clicking solely for visual feedback.
Future survey target:
Inspect whether screenshot creation causes an accessible change in:
- thumbnail count;
- report-image page count;
- current image index;
- or status text.

------------------------------------------------------------
Timing-state interpretation
------------------------------------------------------------
Current logic does not read application state.
It infers the intended operation from the time since the previous press:
- long interval: activate tool;
- short interval: clear corresponding annotation.
This should be treated as established user-interface behavior, not as an error.
A future state-aware implementation is acceptable only if it is more reliable
than the timing heuristic and does not increase latency.
------------------------------------------------------------
Safety and performance requirements
------------------------------------------------------------
- Do not activate a wrong tool if the target control cannot be uniquely found.
- Avoid modal feedback.
- Keep execution latency low enough for mouse-side-button use.
- Restore the original pointer location.
- Do not merge the SUV and arrow clear controls merely because MedEx can be
  configured to clear all annotations from either button.
- Preserve tool-specific clearing unless the user explicitly changes this
  preference.

------------------------------------------------------------
Target: Copy line measurements
------------------------------------------------------------
Name:
复制直线测量值
ControlType:
Button
Migration recommendation:
- Find the exact Button by Name="复制直线测量值".
- Confirm parent context-menu relationship.
- Invoke using InvokePattern.
Clipboard behavior:
- No measurement line: no new usable measurement string.
- One line: one value.
- Two lines: two values separated by "×".
- Three lines: three values separated by "×".
- Four or more lines: only the first three are returned.
Output order:
1. 长径
2. 短径
3. 上下径
Important:
The output order follows measurement creation order and should not be
re-sorted automatically.
------------------------------------------------------------
Clipboard-detection requirement
------------------------------------------------------------
Because "复制直线测量值" may produce no output when no line exists, the
implementation should not inspect only the final clipboard contents.
Recommended method:
1. Preserve ClipboardAll.
2. Replace the clipboard with a unique sentinel or clear it.
3. Invoke the target control.
4. Wait for a clipboard sequence-number change or ClipWait timeout.
5. Distinguish:
   - updated with valid measurement text;
   - updated with unexpected text;
   - no clipboard update;
   - invocation failure.
6. Restore the original clipboard after the caller has consumed the result.
For SUVMax:
- "SUVMax: 0.000" is a valid updated clipboard result.
For line measurements:
- no update should map to NoMeasurement rather than ParseError.
------------------------------------------------------------
Survey conclusion
------------------------------------------------------------
These context-menu actions are strong UIA migration candidates because:
- their controls expose exact, semantically meaningful Names;
- they are standard Button elements;
- the expected result is observable through the clipboard;
- no absolute coordinate dependency is required.

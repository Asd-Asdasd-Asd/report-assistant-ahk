============================================================
Action ID:
copy-report-text-to-image-caption-and-advance
Category:
Dual-monitor MedEx report-image annotation workflow
Hotkey:
Shift + Alt + S
Clinical and reporting purpose:
Attach a selected sentence from the PET/CT report to the currently displayed
report image as its printed image caption.
The caption entered below the image is later shown underneath that image in the
printed PET/CT report.
Workflow context:
- The primary monitor contains the report editing interface.
- Clicking the "报告图像" function opens the saved report-image collection on
  the secondary monitor.
- The user selects a sentence in the report before triggering the hotkey.
- The currently displayed report image on the secondary monitor is the target.
Current implementation:
+!s::
{
    CoordMode "Mouse", "Screen"
    MouseGetPos &xpos, &ypos
    SendInput("^c")
    MouseClick "left", 2821, 1363
    SendInput("^v")
    MouseMove 2884, 704
    SendInput("{WheelDown}")
    MouseMove xpos, ypos
}
Detailed semantics:
Step 1:
Remember the original mouse position.
Current implementation:
MouseGetPos &xpos, &ypos
Purpose:
Allow the workflow to operate on the secondary monitor without leaving the
user's pointer there afterward.
Migration decision:
Retain this behavior.
Step 2:
Copy the currently selected report sentence.
Current implementation:
SendInput("^c")
Precondition:
The required sentence must already be selected in the report editor on the
primary monitor.
Purpose:
Place the selected report description onto the clipboard.
Risk:
The operation depends on the report editor still having keyboard focus and a
valid text selection.
Step 3:
Activate the image-caption text field on the secondary monitor.
Current implementation:
MouseClick "left", 2821, 1363
Purpose:
Focus the text field below the currently displayed image.
Business meaning:
This text becomes the image caption printed underneath the corresponding PET/CT
report image.
Migration strategy:
Prefer locating the caption editor through UIA and setting focus directly.
If the editor supports ValuePattern, consider setting the value explicitly.
Otherwise focus the editor and paste.
Step 4:
Paste the copied report sentence.
Current implementation:
SendInput("^v")
Purpose:
Insert the selected report text into the image-caption field.
Migration strategy:
Retain paste behavior if rich/plain-text behavior is correct.
Consider explicit text assignment only after confirming that it triggers the
same MedEx save/update behavior.
Step 5:
Move focus out of the caption editor.
Current implementation:
MouseMove 2884, 704
Observed intended behavior:
Move the pointer to the center area of the displayed image so that the caption
field can lose focus before advancing.
Important note:
The current code contains MouseMove rather than MouseClick at this step.
The intended workflow description says the image center is clicked to move
focus out of the text field. This discrepancy should be verified against the
actual legacy behavior before migration.
Expected production behavior:
Explicitly remove focus from the caption editor or invoke the application's
caption-save/commit behavior.
Step 6:
Advance to the next report image.
Current implementation:
SendInput("{WheelDown}")
Purpose:
Display the next saved report image so the user can repeat the captioning
workflow.
Migration strategy:
First determine whether the image collection exposes:
- a next-page button;
- a page index control;
- a scrollable image container;
- or only mouse-wheel navigation.
Prefer an explicit "下一页" or next-image control if it reliably corresponds to
the report-image collection. Otherwise retain wheel navigation after anchoring
the pointer over the correct image viewer.
Step 7:
Restore the original pointer position.
Current implementation:
MouseMove xpos, ypos
Purpose:
Return control to the user's original primary-monitor working position.
Migration decision:
Retain.
Current limitations:
- Uses absolute dual-monitor coordinates.
- Assumes a fixed monitor arrangement and resolution.
- Assumes the caption input field remains at a fixed position.
- Assumes wheel input is routed to the image viewer.
- Does not validate that Ctrl+C produced non-empty text.
- Does not verify that the caption was accepted.
- Overwrites the user's existing clipboard content without restoring it.
- The code uses MouseMove at the image center, while the intended behavior is
  described as clicking the image center; this requires confirmation.
Recommended production architecture:
1. Capture and preserve ClipboardAll.
2. Copy the selected report text.
3. Abort if no usable text is obtained.
4. Locate the secondary report-image window.
5. Locate and focus the image-caption editor.
6. Paste or assign the caption.
7. Explicitly commit the caption by focus change or supported control action.
8. Advance to the next report image.
9. Restore the original mouse position.
10. Restore the original clipboard in a finally-style cleanup block.
Safety requirements:
- Run only when the expected MedEx report and report-image windows are present.
- Abort rather than clicking if the target caption field cannot be identified.
- Do not send wheel input until the pointer or focus is confirmed over the image
  viewer.
- Preserve the user's original clipboard even if an intermediate step fails.
Status:
Business workflow documented.
UIA survey of the caption field and next-image mechanism remains pending.
============================================================
Clarification: Caption commit and image advance behavior
============================================================
Related action:
copy-report-text-to-image-caption-and-advance
Validated behavior:
The legacy code intentionally uses MouseMove rather than MouseClick after
pasting into the image-caption editor.
Confirmed sequence:
1. Paste text into the image-caption field.
2. Move the mouse pointer out of the caption field and over the image area.
3. Send WheelDown.
4. The viewer advances to the next report image.
A click on the image is not required.
Interpretation:
- Moving the pointer outside the caption editor is sufficient for routing the
  subsequent mouse-wheel event to the image viewer.
- The wheel event then advances to the next image.
- The previous implementation was intentionally optimized to avoid an
  unnecessary mouse click.
Migration requirement:
Preserve this no-click behavior unless a more explicit and equally reliable
UIA-based next-image action is validated.
Do not insert an additional MouseClick merely to imitate focus loss.
Production options:
1. Preferred if validated:
   Invoke a reliable next-image control directly after updating the caption.
2. Compatible fallback:
   Move the pointer to a validated point inside the image viewer, send
   WheelDown, and then restore the original pointer position.
Open technical question:
It remains useful to determine whether the caption is committed immediately on
paste, on pointer exit, or during the subsequent wheel-driven image change.
This distinction is not required to preserve the existing workflow, but it may
matter if the workflow is later converted to direct UIA value assignment.

Action family: Activate or clear viewer annotations
============================================================
Action IDs:
- toggle-suv-roi-tool
- toggle-arrow-annotation-tool
Category:
High-frequency MedEx viewer annotation automation
Input device:
Two mouse side buttons, exposed to AHK as:
- Ctrl + Win + Shift + M
- Ctrl + Win + Shift + A
Usage priority:
These are among the most frequently used viewer actions, second only to the
screenshot command.
------------------------------------------------------------
SUV ROI measurement workflow
------------------------------------------------------------
Hotkey:
Ctrl + Win + Shift + M
Clinical purpose:
Activate the SUV 3D ROI measurement tool so the user can draw an ROI on the
image and visually read the automatically displayed SUVmax value.
User workflow:
1. Press the mouse side button.
2. The automation activates the SUV 3D tool.
3. Draw an ROI on the image.
4. Read and mentally retain the displayed SUVmax.
5. Press the same mouse side button again.
6. The automation switches to the relevant tool tab and clears the SUV
   measurement annotation.
Current implementation:
^#+m::
{
    CoordMode "Mouse", "Screen"
    startTime := A_TickCount
    timeSinceLastPress := startTime - LastPressSUVTime
    if (timeSinceLastPress > 3000) {
        MouseGetPos &xpos, &ypos
        MouseClick "left", 2471, 826
        MouseMove xpos, ypos
    } else {
        MouseGetPos &xpos, &ypos
        MouseClick "left", 1963, 627
        MouseClick "left", 2043, 1113
        MouseMove xpos, ypos
    }
    global LastPressSUVTime := startTime
    return
}
Behavior:
- If more than 3000 ms have elapsed since the previous press:
  activate the SUV 3D ROI tool.
- If pressed again within 3000 ms:
  switch to the corresponding tool tab and clear the SUV measurement marker.
Known coordinates:
- Activate SUV 3D tool: (2471,826)
- Switch to SUV-related tab before clearing: (1963,627)
- Clear SUV marker: (2043,1113)
Reason for time-based branching:
No reliable state-detection mechanism has yet been identified for determining:
- whether the SUV tool is currently active;
- whether an SUV ROI has been drawn;
- or whether a measurement annotation is present.
The timing rule is therefore an intentional input-design mechanism used to
reuse one mouse button for both activation and clearing.
Do not remove the timing behavior during migration unless an explicit and
reliable state signal is first validated.

------------------------------------------------------------
Why the SUV and arrow clear sequences differ
------------------------------------------------------------
The SUV and arrow tools have separate clear buttons.
Although MedEx configuration can apparently bind both clear actions so that
either clear button removes all annotation types, the current workflow
deliberately uses the tool-specific clear button for each annotation type.
For both tools, the clearing branch contains two operations:
1. Switch to the tab containing the relevant clear control.
2. Invoke the clear button for that annotation type.
The extra tab-selection click is therefore required by the current interface
layout and is not accidental duplication.
------------------------------------------------------------
Shared implementation characteristics
------------------------------------------------------------
Both workflows:
- preserve the original mouse position;
- move to the secondary viewer controls;
- activate or clear the relevant annotation;
- restore the pointer position;
- use timing as a proxy for user intent;
- are bound to dedicated mouse side buttons;
- are highly latency-sensitive and should remain fast.
Migration strategy:
Evaluate each target individually.
Preferred order:
1. Use UIA Invoke for exposed activation and clear buttons.
2. Use relative tab-strip clicks if the individual tab buttons are not exposed.
3. Preserve absolute coordinates only as temporary fallback.
4. Preserve the current mouse-position restoration behavior.
Recommended architecture:
Implement one shared timed dual-action helper with per-tool configuration.
Conceptual structure:
HandleTimedViewerTool(toolPreset)
Preset fields:
- actionId
- activationTimeoutMs
- activateControl
- clearTab
- clearControl
Suggested presets:
SUV:
- timeoutMs: 3000
- activate: SUV 3D button
- clearTab: SUV measurement tab
- clearControl: SUV-specific clear button
Arrow:
- timeoutMs: 1000
- activate: arrow annotation button
- clearTab: arrow annotation tab
- clearControl: arrow-specific clear button
Important behavior:
Do not replace the two timeout values with one shared value without testing.
They encode the user's established operating rhythm:
- SUV: 3000 ms
- Arrow: 1000 ms
Potential future improvement:
If MedEx exposes a reliable active-tool state or annotation-presence state,
the timing heuristic could be replaced or supplemented by explicit state
detection.
Until then, the timing heuristic should remain the production behavior.
Status:
Business semantics documented.
Buttons are reported as UIA-exposed.
Detailed properties and Invoke support remain to be collected in a later
targeted diagnostic pass.

- A value of 0.000 should be treated as a valid command response but generally

Recommended production flow:

2. Clear or mark the clipboard before invocation.
3. Invoke the Button named "复制SUVMax值".
4. Wait briefly for clipboard update.
5. Parse the returned value.
6. Return a structured result:
   - commandSucceeded

Action: Copy line measurement values
------------------------------------------------------------
Accessible control:
Name: 复制直线测量值
ControlType: Button
Observed behavior with no measurement line:
- No visible paste result.
- The clipboard does not appear to receive a new measurement string.
Observed outputs:
One line:
2.6cm
Interpretation:
- Long axis only.
Two lines:
3.0cm×2.6cm (长径×短径)
Interpretation:
- First value: long axis
- Second value: short axis
Three lines:
2.6cm×0.9cm×1.2cm (长径×短径×上下径)
Interpretation:
- First value: long axis
- Second value: short axis
- Third value: craniocaudal dimension
Four lines:
6.2cm×1.2cm×0.6cm (长径×短径×上下径)
Observed rule:
- The fourth line is ignored.
- The copied output contains at most three dimensions.
General output semantics:
- 0 lines: no clipboard measurement output
- 1 line: 长径
- 2 lines: 长径×短径
- 3 lines: 长径×短径×上下径
- 4 or more lines: only the first three values are returned
Important limitation:
The software does not automatically determine which line is anatomically the
longest or shortest. The semantic interpretation depends on the order in which
the user draws the lines.
User workflow requirement:
The user should draw measurements in the intended order:
1. Long axis
2. Short axis
3. Craniocaudal dimension
Implementation requirement:
Do not sort the returned values numerically unless explicitly requested.
Preserve the order emitted by MedEx.
Recommended production flow:
1. Preserve the original clipboard.
2. Clear or mark the clipboard before invocation.
3. Invoke the Button named "复制直线测量值".
4. Wait for clipboard update.
5. If no update occurs within the timeout:
   return NoMeasurement.
6. Parse one to three values separated by the multiplication sign "×".
7. Preserve original order.
8. Return a structured result:
   - dimensionCount
   - longAxis
   - shortAxis
   - craniocaudalAxis
   - rawText
9. Restore the original clipboard when appropriate.
Parser examples:
Input:
2.6cm
Result:
- dimensionCount: 1
- longAxis: 2.6 cm
Input:
3.0cm×2.6cm
Result:
- dimensionCount: 2
- longAxis: 3.0 cm
- shortAxis: 2.6 cm
Input:
2.6cm×0.9cm×1.2cm
Result:
- dimensionCount: 3
- longAxis: 2.6 cm
- shortAxis: 0.9 cm
- craniocaudalAxis: 1.2 cm

Recommended architecture
------------------------------------------------------------
Create one context-menu measurement module with two independent actions:
CopySUVMax()
CopyLineMeasurements()
Shared responsibilities:
- confirm the expected MedEx viewer context menu is open;
- locate the intended Button by exact accessible Name;
- invoke the control;
- detect clipboard change;
- parse the result;
- preserve or restore the original clipboard;
- return explicit success, empty, or invalid-result states.
Do not combine SUV and line-measurement parsing into one generic untyped string
handler.

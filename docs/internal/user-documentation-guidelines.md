# 用户文档原则

本文档用于指导麦旋风的用户说明、更新说明和界面提示。目标用户是没有技术背景的医生，维护者就在附近，沟通成本很低。

## 唯一目标

用户文档只解决三件事：

1. 让用户知道麦旋风能帮助完成什么。
2. 让用户能够顺利开始使用当前功能。
3. 遇到问题时，知道先停下来并直接联系维护者。

文档不是产品宣传、技术展示或行业规范模拟。不能因为功能已经实现，就假设用户自然知道从哪里打开、如何操作或操作后会发生什么。

## 写作方式

- 使用医生容易理解的日常说法，简短、直接、友好。
- 先写“能做什么”和“怎么做”，不要先解释原理。
- 每个步骤都应是具体动作，例如“双击麦旋风图标”或“在设置中点击保存”。
- 只描述当前版本中用户真正能看到、能操作、需要知道的内容。
- 默认保持文档和界面精简；只有不说明就容易用错时，才补充一句必要提示。
- 必要的即时说明优先放在对应界面附近，不要求用户先阅读长文档。
- 语气不必正式。出现问题时直接说“先不要继续操作，截个图并叫维护者来看”即可。

## 默认不写

- 内部实现、配置结构、模块名、错误码和其他技术术语。
- 冗长的原理、完整边界条件或开发历史。
- 产品式 FAQ、反馈表、反馈邮件、工单流程或客服话术。
- 离用户很远的未来规划和尚未开放的能力。
- 为显得完整而增加的重复说明、免责声明和大段格式化内容。

## 发布前快速检查

- 医生能否迅速看懂这个功能解决什么问题？
- 第一次使用的人能否找到入口并完成操作？
- 是否写了用户不需要理解的技术内容？
- 是否把一个简单动作解释得过于复杂？
- 遇到异常时，是否明确要求停止重复尝试并直接联系维护者？

如果一段内容不能降低使用阻力，应删除、缩短，或改为只在相关界面中显示。
